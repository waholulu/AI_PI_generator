# Auto-PI 下一步开发计划

## 1. 计划目标

下一阶段开发不应直接追求“大而全”，而应先把项目从“可演示原型”推进到“可稳定迭代的工程骨架”。建议按下面四个阶段推进：

1. 先修复运行稳定性
2. 再打通模块接口
3. 再补强真实能力
4. 最后完善 UI、测试和文档

## 2. 开发优先级总览

### P0：先解决当前阻塞问题

这些问题不先修，后续开发会持续返工。

#### 2.1 修复 LangGraph checkpoint 导入问题

现状：

- 测试在收集阶段就因 `SqliteSaver` 导入失败而中断

建议动作：

- 核对当前 `langgraph` 与 `langgraph-checkpoint-sqlite` 的实际 API
- 修正 `agents/orchestrator.py` 中的导入与初始化方式
- 让 `build_orchestrator()` 和现有测试至少能正常 import

验收标准：

- `pytest` 可以完成收集
- `e2e_simple_test.py` 至少能运行到工作流中断点

#### 2.2 补齐依赖声明

现状：

- `ui/app.py` 使用 `streamlit`
- `pyproject.toml` 未声明 `streamlit`

建议动作：

- 把 `streamlit` 加入依赖
- 重新校验是否还缺少其他运行时依赖

验收标准：

- 新环境安装依赖后可以直接启动 CLI 和 UI

#### 2.3 统一环境变量加载

现状：

- `main.py` 会加载 `.env`
- `ui/app.py` 没有显式加载 `.env`

建议动作：

- 抽出一个统一的配置加载模块，例如 `config/runtime.py`
- CLI、UI、测试都通过同一入口加载环境变量和默认模型名

验收标准：

- 从命令行和 UI 启动时表现一致

## 3. 第一阶段：工程骨架稳定化

### 3.1 统一状态模型

当前问题：

- `ResearchState` 定义了多个字段
- 但测试和运行代码并不总是完整提供这些字段
- `TypedDict` 语义与实际使用方式不完全一致

建议动作：

- 把状态定义改成更清晰的“可选字段 + 默认值”风格
- 明确每个节点必须读取和写入哪些键
- 写一份状态流转说明

### 3.2 统一模块输入输出契约

当前问题：

- `research_plan.json` 的 `data_sources` 示例字段是 `source`、`data_type`
- `data_fetcher_agent.py` 期待的是 `name`、`api_endpoint`

建议动作：

- 为 `research_plan.json` 定义唯一 schema
- ideation 输出必须严格符合后续模块需要
- literature、drafter、data_fetcher 都按统一 schema 消费

建议优先统一的字段：

- `project_title`
- `keywords`
- `data_sources`
- `methodology`
- `topic_screening`
- `outcomes`
- `exposures`

验收标准：

- 一个由 ideation 生成的 plan 可以被后续所有模块无兼容补丁地消费

### 3.3 抽离公共工具层

建议新增公共模块，减少重复代码：

- OpenAlex 配置与请求封装
- 路径与输出目录初始化
- JSON 读写工具
- 环境变量与模型配置
- 重试和日志工具

收益：

- 降低重复逻辑
- 便于统一错误处理
- 后续扩展其他数据源更容易

## 4. 第二阶段：把“原型能力”升级成“真实能力”

### 4.1 强化 Ideation 模块

目标：

- 从”生成研究想法”升级到”稳定产出可执行研究计划”

**已实现 — Idea Validator（`agents/idea_validator_agent.py`）：**

Pipeline 中新增 `idea_validator` 节点（位于 ideation 和 HITL checkpoint 之间），实现：

1. **原创性验证**：用 LLM 将 idea 转为精准搜索 query → 调用 OpenAlex 搜索近 2 年论文 → LLM 逐篇比对判定重叠度（novel / partially_overlapping / already_published）
2. **数据可用性验证**：将 idea 声称的 data_sources 与 `prompts/data_source_registry.json`（45 个公开数据源）做模糊匹配（difflib, threshold 0.6）
3. **自动替补**：验证失败的 idea 从 `backup_candidates` 池中按分数递补（最多 2 轮），自动更新 `topic_screening.json` 和 `research_plan.json`
4. **记忆写回**：失败 idea 通过 MemoryRetriever 写入 memory，避免重复生成

配置项：`NOVELTY_CHECK_YEARS`、`NOVELTY_QUERIES_PER_IDEA`、`NOVELTY_RESULTS_PER_QUERY`、`DATA_REGISTRY_FUZZY_THRESHOLD`、`VALIDATION_MAX_SUBSTITUTIONS`

待补充动作：

- 对 `passed_gates` 增加可追溯证据字段
- 增加 plan schema 校验失败后的自动重试或降级策略

验收标准：

- 每次运行都能落盘结构稳定、字段一致的 `research_plan.json`
- `output/idea_validation.json` 包含每个 idea 的 novelty verdict + data source 验证状态

### 4.2 强化 Literature 模块

目标：

- 从“少量 OpenAlex 结果卡片”升级到“可复核文献仓储”

建议动作：

- 接入 `arxiv`
- 接入 `Crossref`
- 接入 `Unpaywall`
- 做 DOI 去重
- 丰富 `references.bib`
- 增加 `output/literature_index.json`
- 让 evidence card 至少包含：
  - 研究问题
  - 数据来源
  - 方法
  - 关键结论
  - 局限

验收标准：

- 在一个测试领域下可稳定产出多来源、可追溯的文献索引

### 4.3 强化 Drafter 模块

目标：

- 从“调用一次模型写草稿”升级到“有结构约束的学术初稿器”

建议动作：

- 重写 `prompts/academic_drafter.txt`
- 强制固定章节结构
- 只允许引用 `references.bib` 中存在的条目
- 草稿中显式加入 AI 辅助声明
- 对文献证据和 plan 做分层拼装，而不是简单字符串拼接

验收标准：

- 输出的 `Draft_v1.md` 至少结构完整、引用来源明确、失败时降级清晰

### 4.4 强化 Data Fetcher 模块

目标：

- 从 mock 下载器升级为真实公开数据采集器

建议动作：

- 先不要直接做复杂 Agent 沙箱
- 第一版优先做“确定性适配器”：
  - Census API
  - OpenStreetMap / Overpass
  - World Bank Open Data
- 每个适配器负责：
  - 预检查端点可达性
  - 下载
  - 转换为 Parquet 或 GeoParquet
  - 写入 manifest

这样做的原因：

- 工程可控
- 调试成本低
- 更容易测试
- 后续再升级成 Agent 自主抓取时也能复用适配器层

验收标准：

- 至少 1 到 2 个公开数据源可以被真实下载并写入规范 manifest

## 6. 第四阶段：测试与质量保障

### 6.1 重构测试分层

建议把测试拆成三层：

1. 纯单元测试
2. 模块集成测试
3. 端到端测试

建议重点：

- 单元测试尽量不依赖真实外部 API
- 用 mock 或 fixture 固定 OpenAlex / LLM / 数据源返回
- 只保留少量人工触发的真实联网测试

### 6.2 加入最关键的回归测试

优先补这些用例：

- orchestrator 构图成功
- ideation 输出 schema 合法
- literature 能处理空关键词和空结果
- drafter 在无 API key 时能正确 fallback
- data_fetcher 能写出合法 manifest
- **memory + field_scan 一体化回归测试**：`tests/test_memory_and_field_scan.py` 覆盖两种场景——存在历史 CSV 记忆且 OpenAlex 返回正常结果时，可以生成包含 `keywords.high_traction` 的 `field_scan.json`；以及记忆与扫描结果均为空时，仍能生成最小 `field_scan.json` 且字段存在不报错。

### 6.3 增加“契约测试”

这是当前项目非常值得做的一类测试。

建议增加：

- `research_plan.json` schema 契约测试
- `run_index.json` 字段完整性测试
- literature card 基本字段测试

## 7. 推荐执行顺序

如果只做一轮短周期开发，建议顺序如下：

1. 修复 checkpoint 导入问题
2. 补齐 `streamlit` 依赖与统一配置加载
3. 统一 `research_plan.json` schema
4. 重构 ideation 到 literature 到 data_fetcher 的接口
5. 把 data fetcher 先做成确定性适配器版本
6. 重写 academic drafter prompt
7. 重构测试并补 README

## 8. 一句话结论

这个项目的下一步重点不是继续扩功能面，而是先把“能跑的原型”修成“接口稳定、依赖完整、测试可用”的工程底座；只有这样，后续的文献增强、真实数据抓取和更强的多智能体能力才值得继续投入。
