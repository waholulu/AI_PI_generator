# Auto-PI 项目现状与关键设定

## 1. 项目当前定位

本项目是一个基于 Python + LangGraph 的多智能体研究脚手架原型，目标是把科研前期的四个环节串成流水线：

1. 领域扫描
2. 选题与研究设计
3. 文献收集
4. 初稿生成
5. 数据采集后停机

当前代码已经具备一条从命令行输入研究领域，到生成研究计划、文献索引、草稿和模拟数据清单的基础链路，但整体仍处于“可演示原型”阶段，距离规划书中的完整版本还有明显差距。

## 2. 当前已实现的核心功能

### 2.1 入口与编排

- 入口文件是 `main.py`
- 启动时会读取 `.env`
- 用户输入一个宏观研究领域描述
- 调用 `agents/orchestrator.py` 构建 LangGraph 工作流
- 工作流节点顺序为：
  - `field_scanner`
  - `ideation`
  - `literature`
  - `drafter`
  - `data_fetcher`
- 在 `ideation` 之后、`literature` 之前设置了人工中断点

### 2.2 Field Scanner

对应文件：`agents/field_scanner_agent.py`

已实现内容：

- 读取 `OPENALEX_API_KEY` 和 `OPENALEX_EMAIL`
- 使用 OpenAlex 检索领域相关高被引论文
- 提取标题、被引数、年份、作者、概念标签
- 输出 `output/field_scan.json`

当前特点：

- 只有 OpenAlex 单一来源
- 输出较轻量，主要用于后续 ideation 约束

### 2.3 Ideation / Research Design

对应文件：`agents/ideation_agent.py`

已实现内容：

- 读取 Field Scan 结果
- 从本地 SQLite 记忆库中检索历史选题上下文
- 使用 Gemini Fast 模型生成候选题
- 依据 `passed_gates` 区分通过与淘汰题目
- 输出：
  - `output/topic_screening.json`
  - `output/ideas_graveyard.json`
  - `output/topic_ranking.csv`
- 使用 Gemini Pro 模型生成最终 `config/research_plan.json`

当前特点：

- 已具备“候选题生成 -> 筛选 -> 排序 -> 最终计划”主流程
- Pydantic schema 已经定义了比较严格的定量研究字段
- 但候选题和最终 plan 的 schema 仍然偏宽松，和后续模块接口尚未完全对齐

### 2.4 Literature Harvester

对应文件：`agents/literature_agent.py`

已实现内容：

- 读取 `config/research_plan.json` 的 `keywords`
- 用 OpenAlex 检索少量论文
- 为每篇论文生成一张简化证据卡
- 输出：
  - `data/literature/cards/*.json`
  - `data/literature/index.json`
  - `output/references.bib`

当前特点：

- 目前只做 OpenAlex 检索
- 没有真正下载 PDF
- 没有 arXiv、Crossref、Unpaywall 的回退链
- BibTeX 生成逻辑较简化

### 2.5 Academic Drafter

对应文件：`agents/drafter_agent.py`

已实现内容：

- 读取 `config/research_plan.json`
- 读取 `data/literature/index.json`
- 读取提示词文件 `prompts/academic_drafter.txt`
- 调用 Gemini 生成 `output/Draft_v1.md`
- 调用失败时写入 fallback 草稿

当前特点：

- 具备最基本的“计划 + 文献证据 -> 草稿”能力
- 但 prompt 仍是占位内容，当前文件只有 `Test prompt`
- 没有引用一致性校验，也没有更细的章节级生成逻辑

### 2.6 Data Fetcher

对应文件：`agents/data_fetcher_agent.py`

已实现内容：

- 读取 `research_plan.json` 中的 `data_sources`
- 遍历数据源并写入 `data/raw/manifest.json`
- 生成模拟 `.parquet` 文件
- 写入 `output/run_index.json`
- 将状态标记为 `completed`

当前特点：

- 当前是 mock 实现，不是真实采集器
- 虽然初始化了 Anthropic 客户端，但没有真正使用到下载代理能力
- 已经保留了“采集完成即停机”的边界意识

### 2.7 本地记忆库

对应文件：`agents/memory_retriever.py`

已实现内容：

- 使用 SQLite 初始化 `data_vault/vault_index.sqlite`
- 建立 `idea_vault` 表
- 支持 `store_idea()`
- 支持按领域关键词检索历史上下文 `retrieve_domain_context()`

当前特点：

- 检索能力已存在
- 主流程里已经使用“读取历史记忆”
- 但主流程没有把新生成的候选题和最终结果系统性写回数据库，因此长期记忆闭环尚未真正形成

### 2.8 UI 监控界面

对应文件：`ui/app.py`

已实现内容：

- 基于 Streamlit 提供基本监控页面
- 可以输入研究领域并启动/恢复工作流
- 可查看当前状态和下一节点
- 在 `literature` 前暂停时，可展示 plan 并允许点击继续

当前特点：

- 已实现最小可用监控/审批界面
- 但依赖未在 `pyproject.toml` 中声明 `streamlit`
- UI 本身没有显式加载 `.env`

## 3. 关键设定

### 3.1 运行环境

- Python 版本固定为 `3.10.x`
- 配置位于 `pyproject.toml` 和 `.python-version`

### 3.2 关键依赖

`pyproject.toml` 中已声明的关键依赖包括：

- `langgraph`
- `langgraph-checkpoint-sqlite`
- `langchain-google-genai`
- `langchain-anthropic`
- `pyalex`
- `python-dotenv`
- `pandas`
- `geopandas`
- `pyarrow`
- `pymupdf`
- `arxiv`
- `scholarly`
- `semanticscholar`

注意：

- `ui/app.py` 使用了 `streamlit`，但当前未写入依赖

### 3.3 环境变量

从代码和测试可确认的关键环境变量包括：

- `GEMINI_API_KEY`
- `GOOGLE_API_KEY`
- `GEMINI_FAST_MODEL`
- `GEMINI_PRO_MODEL`
- `OPENALEX_API_KEY`
- `OPENALEX_EMAIL`
- `UNPAYWALL_EMAIL`

隐含依赖：

- `data_fetcher_agent.py` 实际上还依赖 Anthropic 侧配置，但仓库内没有清晰文档化

### 3.4 工作流状态与通信方式

项目采用“文件落盘 + 状态指针”的方式串联模块：

- LangGraph `ResearchState` 里主要传递路径和状态
- 模块之间核心依赖文件是 `config/research_plan.json`
- 关键产物主要落在：
  - `output/`
  - `data/literature/`
  - `data/raw/`

### 3.5 当前默认输出物

一次完整流程理论上会生成：

- `output/field_scan.json`
- `output/topic_screening.json`
- `output/ideas_graveyard.json`
- `output/topic_ranking.csv`
- `config/research_plan.json`
- `data/literature/cards/*.json`
- `data/literature/index.json`
- `output/references.bib`
- `output/Draft_v1.md`
- `data/raw/manifest.json`
- `output/run_index.json`

### 3.6 测试现状

`tests/` 中已经有：

- `test_field_scanner.py`
- `test_ideation.py`
- `test_literature.py`
- `test_drafter.py`
- `test_data_fetcher.py`
- `e2e_simple_test.py`
- `test_api_keys.py`

但当前测试并未稳定通过。根据 `test_output.txt`，测试收集阶段就出现错误：

- `agents/orchestrator.py` 中 `from langgraph.checkpoint.sqlite import SqliteSaver` 无法导入

这意味着：

- 当前环境与代码的 checkpoint 用法存在不匹配
- 测试体系还不能作为可靠回归保障

## 4. 当前实现与规划书的主要差异

规划文档 `Auto-PI_研究脚手架系统计划书.md` 描述的是较完整的目标形态；现有代码只实现了其中的最小原型。

已经落地的方向：

- LangGraph 串行 DAG
- 研究计划作为中心配置文件
- OpenAlex 接入
- Gemini 生成候选题与研究计划
- Streamlit 监控雏形
- 数据采集完成即停止的边界设定

尚未完整落地的关键能力：

- 文献多源检索链路
- PDF 获取与证据片段提取
- 真正可执行的数据采集代理
- 长期记忆写回机制
- 更完整的 Prompt 工程
- 依赖与运行环境的稳定化
- 可持续通过的自动化测试

## 5. 当前阶段判断

综合来看，这个仓库已经不是空壳，而是一个“主链路可见、关键边界已定义、但多数模块仍是简化实现”的研究自动化原型。最值得优先处理的不是继续堆新功能，而是先把运行时稳定性、模块接口一致性、依赖声明和测试体系修好，再进入下一轮增强开发。
